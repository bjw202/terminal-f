# ADR-007: 자동화 규칙 엔진 (M2.1)

상태: 채택 (2026-07-02)

## 배경

M2.0에서 게이트가 적용된 주입 경로를 구축했다. M2.1은 첫 번째 *자동*
트리거를 추가한다: git 저장소를 감시하다가 워킹 트리가 변경되면, 라벨이 지정된
페인으로 프롬프트 주입을 제안하는 것이다(로드맵 §2.1의 "git-diff → 리뷰 요청"
시나리오). 이는 외부 브로커 API(M2.2)에 앞서 채택된 첫 단계다.

## 결정

### 파일시스템 알림이 아닌 폴링

자동화 스레드는 활성화된 각 규칙을 `POLL_INTERVAL_MS`(2초)마다 폴링하며,
`git -C <repo> status --porcelain` + `diff --stat`를 외부 프로세스로 실행한다.

- `notify`/inotify 방식의 FS 감시는 기각했다: 목표 지연 시간이 초 단위인
  기능을 위해, 플랫폼별 특이점(Windows의 재귀 감시 의미론, 에디터의
  write-rename 소음)을 가진 무거운 의존성을 추가하게 된다. 로드맵 자체가
  5초 디바운스를 명시했으므로 — 2초 폴링은 그 안에 충분히 들어간다.
- "무엇이 변했는가"의 진실 원천(source of truth)은 git이고, `git status`는
  이미 에디터 소음을 병합해 준다. 이를 폴링하는 것이 원시 FS 이벤트를
  감시해 다시 유도하는 것보다 단순하고 정확하다.
- 비용: 규칙이 존재하고 활성화된 동안에만 2초마다 `git`을 몇 번 실행하는
  정도다. 무시할 수 있는 수준이며 규칙 개수에 의해 상한이 정해진다.

### 결정 로직은 순수 함수이며 단위 테스트된다

`RuleRuntime::decide`는 (규칙, 현재 diff 해시, 현재 시각, 수동 여부)로부터
IO 없이 Fire/Skip을 반환한다. 순서대로 다음을 인코딩한다: 디바운스(두 번의
폴링에 걸쳐 동일한 해시), 중복 제거(마지막으로 발동한 diff에 대해 재발동하지
않음), 쿨다운, 그리고 롤링 60초 발동 빈도 제한(rate limit). git IO와 Tauri
이벤트 발신은 커맨드 계층에 있다. 이렇게 하면 저장소나 실제 시계 없이도
까다로운 로직을 테스트할 수 있다(테스트는 합성 `Instant`와 해시를 전달한다).

diff "해시"는 `git status --porcelain` 출력의 FNV-1a다 — `diff --stat`의
문구와 무관하게 워킹 트리의 파일 집합/상태가 변할 때에만 바뀌므로,
중복 제거가 안정적이다.

### 기본 확인 모드; 주입은 M2.0 경로를 재사용

채택된 로드맵 결정에 따라, 규칙은 명시적으로 `auto`로 설정하지 않는 한
`confirm` 모드다. 발동 시:

- **confirm**: `Proposal`이 저장되고 `automation-proposal` 이벤트가
  발신된다. UI는 승인/기각(Approve/Dismiss) 토스트를 표시한다. 승인은
  `resolve_proposal`을 호출하는데, 이는 수동 주입에 쓰이는 것과 *동일한*
  `do_inject`를 실행한다 — 따라서 킬 스위치, 페인별 허용 목록(allowlist),
  유휴 게이트(idle gate), 브래킷 붙여넣기(bracketed paste), 감사 로그가 모두
  적용된다. 감사 로그의 `source`는 ("manual"이 아닌) 규칙 id이므로, 로그에서
  자동 주입과 수동 주입을 구분할 수 있다.
- **auto**: `do_inject`가 즉시 실행된다(여전히 모든 게이트가 적용된다).

제안(proposal)하기 전에 엔진은 대상을 결정하고 허용 목록을 확인하므로,
승인해도 실행될 수 없는 제안을 표면화하는 일은 없다. 승인 시점에 주입이
실패하면(예: 대상이 잠시 바쁜 경우) 제안은 재시도를 위해 다시 큐에 들어간다.

### 발동 집계

폴링 발동은 `fire_rule`이 성공했을 때에만 *기록*된다(`record_fire` — 중복
제거/쿨다운/발동 빈도 제한을 활성화한다). 대상이 결정되지 않았거나, 허용
목록에 없거나, 바쁜 경우에는 발동이 기록되지 않고 다음 폴링에서 재평가된다 —
따라서 아직 준비되지 않은 대상이 해당 diff에 대한 한 번의 기회를 조용히
소모해 버리는 일이 없다. `run_rule_now`는 디바운스/쿨다운/중복 제거/발동
빈도 제한을 우회하지만(명시적 사용자 행동이므로), confirm/auto, 허용 목록,
유휴 게이트는 여전히 준수한다.

### 영속화

규칙은 `config.json`에 저장된다(`automation`, 스키마 v4; 이전 버전 설정은
빈 목록으로 기본 설정된다 — 자명한 마이그레이션이다). 런타임 상태(디바운스/
쿨다운/발동 빈도 제한 카운터, 대기 중인 제안)는 절대 영속화되지 않는다.

## 트레이드오프/한계

- 2초 폴링은 발동까지 최대 약 2초의 지연에 디바운스 폴링 한 번이 더해짐을
  의미한다. 리뷰 프롬프트에는 충분하지만, 1초 미만의 반응에는 적합하지 않다.
- M2.1에는 git-diff 감시 소스만 존재한다. 페인 출력 소스와 타이머 소스
  (로드맵 §2.1)는 향후 작업이다. `Rule` 구조는 스키마를 깨지 않고 태그된
  `source` enum으로 확장할 수 있다(serde default).
- 대기 중인 제안은 메모리에만 있다: 앱을 재시작하면 승인되지 않은 제안이
  사라진다(수용 가능 — 어차피 낡은 제안이며, 다음 diff가 다시 제안한다).
- 아직 파일 스풀(spool)은 없다(로드맵 §2.5): 출력 스트림 감시자와 느린 외부
  구독자는 M2.2와 함께 온다.

## M2.1.5 부록: 타이머 소스

`Rule.source`는 태그된 enum이 되었다(`gitDiff { repo }` | `timer { everyMs }`).
하위 호환을 위해 선택 필드다 — M2.1.5 이전 규칙(`repo`만 있고 `source`가
없는 경우)은 `effective_source()`를 통해 `gitDiff`로 해석된다. 설정 스키마는
v5가 되었다(추가만 있는 자명한 마이그레이션).

근거(로드맵 §2.7 #1): "N분마다 주입"에는 외부 프로세스도 git도 필요 없다 —
판단이 필요 없는 스케줄링이므로, M2.2 파이프 API가 아닌 인프로세스 엔진에
속한다. 타이머 발동은 `RuleRuntime::decide_timer`를 사용하고(`everyMs`마다
한 번 발동, 첫 발동은 규칙이 관측된 후 한 인터벌 뒤, 핫 루프를 피하기 위한
`TIMER_MIN_MS` 하한 1초), 동일한 발동 빈도 제한 / confirm-vs-auto / 유휴 /
감사 장치를 재사용한다. diff 요약은 git-diff 규칙만 계산한다. 타이머 규칙은
`None`을 전달하며, 그 템플릿에는 채워 넣을 `{{diffStat}}`가 없다.

참고: 내부 태그(internally-tagged) enum의 경우 serde의 컨테이너
`rename_all`은 배리언트 이름만 바꾸고 그 필드는 바꾸지 않으므로, `everyMs`의
와이어 이름은 명시적인 `#[serde(rename)]`으로 지정한다.
`rule_source_wire_format` 테스트가 이를 고정한다.

## 향후 계획 (M2.2)

네임드 파이프 제어 API가 외부 브로커 프로세스에 `listPanes` /
`subscribeOutput` / `injectPrompt`를 노출한다. 이들도 동일한 게이트가 적용된
`do_inject`를 호출하므로, 트리거의 출처와 무관하게 게이트는 백엔드 쪽에 남는다.
