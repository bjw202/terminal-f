# ADR-004: 백프레셔와 링 버퍼 정책

상태: 채택 (M0)

## 결정

- **읽기**: 세션당 블로킹 리더 스레드 1개, 8 KB 단위 읽기, UTF-8 청크
  경계 복구 (불완전한 멀티바이트 꼬리는 다음 청크로 이월, 유효하지 않은
  바이트는 U+FFFD로 치환).
- **버퍼**: 세션별 유한 링 버퍼 — 최대 1 MiB **그리고** 1024 청크
  (둘 중 먼저 도달하는 쪽 적용). push는 절대 블록하지 않는다. 넘치면
  가장 오래된 청크부터 축출하고 드롭 카운터를 증가시킨다. 세션별 단조
  증가 시퀀스 번호(seq) 카운터를 유지한다.
- **발행(emit)**: 단일 이미터 스레드가 **16 ms**마다 틱하며, 활성
  워크스페이스의 세션만 스캔하고, 틱당 페인당 최대 1개의 병합(coalesced)
  `pty-output` 이벤트를 발행한다 (대기 중인 모든 청크를 이어붙임).
  비활성 워크스페이스는 아무것도 발행하지 않는다. 발행은 링에서 pull
  방식으로 가져온다 — 리더→이미터 채널이 없으므로 채널이 가득 찰 일이
  없다.
- **리플레이**: 워크스페이스 전환 시 프론트엔드가
  `replay_pane(paneId, fromSeq)`를 호출해 가져온다. 라이브 발행은
  리플레이 종료 시퀀스 번호(seq)부터 재무장되므로 (`replay_synced`
  플래그), 리플레이와 라이브 이벤트가 뒤섞이거나 중복될 수 없다.
- **드롭 정책: 가장 오래된 것부터 버림(oldest-drop)** (유한 링). 드롭은
  집계되어 해당 페인에 눈에 보이는 `[output overflow…]` 마커로 표시된다.
  완전한 원시 데이터 보존은 명시적으로 주장하지 **않는다** — 그러려면
  아래의 파일 스풀 옵션이 필요하다.

## 청크마다 발행하지 않는 이유

`cmd /c type bigfile` 하나가 초당 수천 개의 청크를 만들 수 있다. Tauri
이벤트 하나하나가 JSON 직렬화 + IPC 홉 + JS 디스패치를 수반한다.
청크 단위 발행은 브리지를 범람시키고 렌더 루프를 굶긴다. 배칭은 출력량과
무관하게 가시 페인당 이벤트 발생률을 약 62/s로 제한한다.

## 오버플로에 대해 검토한 대안

1. **가장 오래된 청크 드롭 (채택)**: 메모리가 상수로 유지되며, 최악의
   경우에도 xterm 스크롤백에 대개 아직 남아 있는 오래된 출력만 잃는다
   (5000줄 스크롤백 vs 1 MiB 링).
2. **하드 백프레셔 (리더 블록)**: 가득 찬 ConPTY 파이프를 통해 자식
   프로세스를 블록한다 — UI가 멈추면 사용자의 셸까지 얼어붙는다. 기각.
3. **파일 스풀 폴백**: 진정한 보존이 가능하지만 M1+로 보류 (디스크
   할당량/정리 문제가 추가됨). `dropped_bytes/dropped_chunks` 카운터가
   스풀을 붙일 수 있는 연결 지점이다.
4. **요약(summarization)**: 출력에 대한 의미론적 이해가 필요하다. 범위 밖.

## 제안된 기본값과의 차이

스펙은 "페인당 최대 대기 청크 ≈ 1024"를 제안했다 — 이를 채택하되 바이트
상한(1 MiB)을 추가했다. 청크 개수만으로는 메모리 상한이 되기 어렵기
때문이다 (청크 하나가 8 KB일 수 있어 최악의 경우 페인당 8 MiB). 16 ms
배칭은 그대로 채택했다.

## 트레이드오프

- 비활성 상태에서 1 MiB 넘게 출력한 페인은 최근 약 1 MiB만 (+ 드롭 마커)
  가지고 돌아온다. M0에서는 허용 가능하다.
- 16 ms 스캔은 유휴 상태에서도 깨어난다. 비용은 맵 조회 몇 번 수준이다
  (soak 벤치마크에서 측정, 무시할 만함).

## 향후 계획

페인별 바이트 예산을 적응형으로 만들 수 있다 (페인이 적으면 링을 더
크게). 워처 기능을 위한 파일 스풀. M1의 대량 출력 렌더링 목표가
요구한다면 xterm.write 콜백을 통한 프론트엔드 흐름 제어.
