# ADR-002: 워크스페이스 keep-alive 정책

상태: 채택 (M0)

## 결정

- 백엔드: 비활성 워크스페이스의 PTY 세션은 손대지 않고 계속 실행한다.
  리더 스레드는 ConPTY를 계속 비운다 (자식 프로세스가 가득 찬 파이프에서
  블록되는 일이 없도록). 출력은 세션별 링 버퍼에 쌓이며, 이벤트는
  발행하지 않는다.
- 프론트엔드: 전환 시 각 페인의 xterm을 시각 스냅샷(visual snapshot)으로
  직렬화하고 (serialize addon, 스크롤백은 1000줄로 제한) xterm 인스턴스는
  **완전히 폐기(언마운트)한다**. 복귀 시: 새 xterm 생성 → 스냅샷 쓰기 →
  자리를 비운 동안 도착한 출력을 위한 `replay_pane(paneId, lastSeq)` 호출 →
  라이브 이벤트 재무장(re-arm).

`display:none`과 언마운트는 절대 혼용하지 않는다: 언마운트가 유일한
정책이다.

## 검토한 대안

1. **비활성 워크스페이스를 `display:none`으로 마운트 유지**: 전환이 가장
   빠르지만 (재마운트 불필요) 숨겨진 모든 페인이 xterm 버퍼 전체, 렌더러
   캐시, DOM을 살려둔다 — 메모리가 워크스페이스 × 페인 수에 선형으로
   증가하여 메모리 예산 우선순위에 정면으로 어긋난다. 또한 숨겨진 WebGL
   캔버스는 컨텍스트 축출(eviction)의 1순위 대상이라 예기치 않은 폴백을
   일으킨다.
2. **언마운트하되 Terminal 객체를 분리된 상태로 유지**: JS 버퍼 메모리를
   어차피 유지하므로 (1)에 비해 절약이 거의 없다. 기각.
3. **스냅샷 없이 리플레이만으로 복원**: 리플레이는 유한한 링에 아직 남아
   있는 것만 복원한다. 오래 방치된 워크스페이스는 빈 화면으로 돌아온다.
   스냅샷은 시각 상태를 저렴하게 (문자열 하나로) 보존한다.

## 근거

스펙이 메모리 예산을 우선시하므로 → 언마운트가 권장 기본값이다.
seq 기반 리플레이가 언마운트로 생기는 공백(언마운트 중 생성된 출력)을
메우고, 스냅샷이 리플레이가 메울 수 없는 부분(유한한 링에서 축출된
출력)을 메운다.

## 트레이드오프

- 전환 비용에 xterm 생성 + 스냅샷 쓰기 + 리플레이가 포함된다
  (BENCHMARK.md에서 M0 목표인 p95 < 150 ms 대비로 측정).
- serialize는 바이트 단위로 정확한 이력이 아니라 *시각적* 재구성이다
  (ADR-003). 스냅샷 상한을 넘는 스크롤백은 전환 시 시각적으로 소실된다.
- M1의 캐시된 전환 p95 < 50 ms 목표를 위해서는 재검토가 필요할 수 있다
  (예: 마운트된 채 숨겨진 워크스페이스 1–2개의 LRU) — M0에서는 의도적으로
  구현하지 않았다.

## 향후 계획

M1 전환 목표가 요구한다면 하이브리드 정책(가장 최근에 사용한
워크스페이스를 마운트 상태로 유지)을 도입한다. 정책이 바뀌면 여기에
기록한다.
