# ADR-001: 백엔드 소유 PTY 세션 레지스트리

상태: 채택 (M0)

## 결정

Rust 백엔드가 PTY 세션의 단일 진실 공급원(single source of truth)인
`SessionRegistry`를 소유한다: `sessions`, `pane_to_session`,
`workspace_to_sessions`, 그리고 출력 이미터가 사용하는 활성 워크스페이스
마커까지 포함한다. 프론트엔드는 프로세스 핸들을 절대 보유하지 않으며,
의미가 백엔드 측에서 정의된 커맨드를 통하지 않고서는 세션을 생성하거나
파괴할 수 없다. 추가로 세션뿐 아니라 **페인 트리 / 워크스페이스 상태**도
백엔드가 소유하도록 하여, 모든 레이아웃 변경이 커맨드가 되고 불변식이
테스트된 한 곳에만 존재하도록 했다.

## 검토한 대안

1. **프론트엔드 소유 세션** (프론트엔드가 백엔드에 "spawn/kill"을 지시하고
   매핑을 JS에 보관): 기각 — WebView 리로드, 예외, HMR이 발생하면
   프로세스가 고아가 되고, 레이아웃 상태와 프로세스 상태 사이의 경쟁
   조건을 관측할 수 없게 되며, 필수 요구사항을 위반한다.
2. **백엔드 세션 + 프론트엔드 소유 레이아웃 트리**: 동작은 가능하지만,
   페인 닫기가 트리 노드 제거*와* 세션 종료를 원자적으로 수행해야 한다.
   트리가 JS에 있으면 2단계 프로토콜이 필요하고 오류 시 상태가 어긋날
   수 있다. 단일 소유자를 택하기 위해 M0에서는 기각했다.
3. **PTY 전용 별도 데몬 프로세스** (tmux 방식 서버): 가장 견고한 방식
   (UI가 크래시해도 세션이 살아남음)이지만 M0 범위 밖이다. 레지스트리
   API는 추후 소켓 뒤로 옮길 수 있는 형태로 설계했다.

## 근거

세션 수명은 webview/DOM 수명과 분리되어야 한다 (워크스페이스 keep-alive
요구사항). 명시적인 맵을 가진 레지스트리를 두면 필요한 수명 주기 규칙
(페인 닫기 → 하나 종료, 워크스페이스 삭제 → 해당 목록 종료, 전환 →
아무것도 종료하지 않음)을 UI 없이도 아주 쉽게 감사하고 테스트할 수 있다.

## 트레이드오프

- 모든 레이아웃 상호작용이 IPC 왕복을 수반한다 (Tauri에서 약 1ms 미만.
  M0의 커맨드 발생 빈도에는 충분하며, 드래그 리사이즈는 과도한 호출을
  피하기 위해 mouseup 시점에만 커밋한다).
- 백엔드가 재시작되면 모든 세션이 죽는다 (아직 데몬 분리가 없음).

## 향후 계획

M1+: 레지스트리를 트레이트 경계 뒤로 추출하여 별도의 장수(long-lived)
프로세스에서 실행할 수 있게 하고, 세션 재생성(respawn)과 cwd 추적을
추가한다.
