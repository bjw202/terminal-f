# ADR-009: 프로젝트 프로필 템플릿 (Phase B)

상태: 채택 (2026-07-02)

## 배경

로드맵 §3: 사전 정의된 다중 분할 워크스페이스(페인, cwd, 페인별 시작
커맨드)를 한 번의 동작으로 구체화하는 "프로젝트 프로필" —
tmuxinator/tmuxp에 해당하는 포지셔닝이다. 기존 페인 모델을 재사용해야 하며,
트리 불변식이나 주입 안전 작업을 약화시켜서는 안 된다.

## 결정

### 템플릿 = 페인 트리 청사진 + 변수 (모델 재사용)

`TemplateNode`는 `PaneNode`(Pane | Split)를 그대로 반영하되, 런타임 상태
(id, sessionId)를 빼고 페인별 `cwd`, `labels`, `allowInjection`,
`allowObserve`, `command`, `startupCommand`와 최상위 `params`를 더한 것이다.
`build_tree`는 변수를 치환하고, 새 id를 할당해, 일반적인 `PaneNode`를
생성한다 — 따라서 동일한 `check_invariants`와 페인 연산이 그대로 적용된다.
새로운 트리 규칙도, 새로운 불변식도 없다.

### `startupCommand` ≠ `command`

- `command`는 셸을 대체한다(커맨드가 종료되면 페인이 죽는다) — 기존 동작이다.
- `startupCommand`는 기본 셸이 준비된 후 그 *안에서* 실행되므로, 커맨드가
  끝난 뒤에도 페인은 살아 있는 셸로 남는다. 이것이 사용자가 기대하는
  tmuxinator 동작이다("워크스페이스를 열고 각 페인에서 `claude`/`npm run dev`를
  시작하되, 셸은 남겨 달라").

실행은 M2.0의 유휴 감지를 재사용한다: 스폰 시 세션별 시작 스레드가 셸이
출력을 낸 뒤 유휴 상태가 될 때까지(≥800 ms, 상한 10초) 기다린 후 커맨드를
타이핑한다. 중요한 점은, `startupCommand`는 외부 주입이 아니라 **사용자
자신의 템플릿 동작**이라는 것이다 — 따라서 `allowInjection`으로 게이트되지
*않고*, 주입 감사 로그에도 기록되지 *않는다*. (외부 자동화는 여전히 둘 다
적용된다.)

### 변수: `${param}`과 `${env:VAR}`

`build_tree`는 params 맵으로부터 `${name}`을(선언된 param은 미지정 시
`default`로 채워진다), 프로세스 환경으로부터 `${env:VAR}`를 치환한다. 알 수
없는 플레이스홀더는 오류를 내는 대신 문자 그대로 남긴다. 템플릿이 적용 실패로
끝나는 대신 눈에 보이게 완만히 저하되도록 하기 위해서다.

### 항상 새 워크스페이스

`apply_template`은 언제나 워크스페이스를 *생성*만 하며(`create_with_root`),
절대 덮어쓰지 않는다 — 덮어쓰면 살아 있는 세션을 파괴해 사용자를 놀라게
한다. 상한(워크스페이스 하드/소프트, 템플릿당 ≤16 페인)은 사전에 강제된다.

### 2계층 저장소 + 워크스페이스 신뢰(trust)

- 전역 템플릿: `<config>/templates/<name>.json` — 사용자가 작성한 것으로
  신뢰된다. `save_template`은 이름에 경로 구분자를 거부해, 쓰기가 templates
  디렉터리를 벗어날 수 없게 한다.
- 저장소 로컬: `<repo>/.terminal-f/profile.json` — 저장소를 통해 공유할 수
  있다. 갓 클론한 저장소가 커맨드를 자동 실행하는 프로필을 담고 있을 수
  있으므로, `read_repo_profile`은 `hasCommands`와 해당 저장소가 이미
  신뢰되었는지를 보고한다. 프론트엔드는 커맨드를 실행하는 신뢰되지 않은
  프로필을 적용하기 전에 **반드시** 명시적 사용자 확인을 받아야 하며, 그 후
  저장소를 `trustedRepos`(설정, 스키마 v7)에 기록해 다시 묻지 않는다. 이는
  VS Code의 워크스페이스 신뢰 게이트를 따른 것이며, ADR-006/008과 같은
  "낯선 이의 커맨드를 실행하지 않는다" 경계다.

## 트레이드오프/한계

- 시작 타이밍은 휴리스틱(출력 안정화 기준)이며, 유휴 게이트(ADR-006)와 같은
  근거와 주의점을 공유한다: 멈춰 있는 셸은 10초 상한 이후 이르게 발동할 수
  있다. 셋업 커맨드에는 수용 가능하다.
- `startupCommand`는 페인 트리에 영속화되므로, 해당 페인의 세션이 (재)스폰될
  때마다 다시 실행된다 — 의도된 동작이지만(재현 가능한 셋업), 부작용이 있는
  커맨드는 새 세션에서 또 실행된다. 문서화해 둔다.
- 템플릿으로 저장하기(save-as-template)는 cwd를 문자 그대로 유지한다
  (`${repo}`로 자동 일반화하지 않음). 파라미터화는 사용자가 JSON을 편집해
  수행한다. 기능을 예측 가능하게 유지하기 위함이다.
- 신뢰는 적용 시점에 확인되고 기억된다. 구체화된 템플릿 워크스페이스는 그
  이후 사용자 자신의 것이다(저장소가 신뢰되었으므로, 그 startupCommand들은
  재시작 시 다시 묻지 않고 재실행된다).

## 테스트

- 단위 (`template.rs`): 변수 치환(param/env/알 수 없는 경우), validate
  (누락된 param, 페인 상한), build_tree(id 할당, 불변식 유지, 라벨 소문자화,
  startup 치환), has_commands, 트리 왕복(round-trip).
- E2E (autotest): `${marker}` param과 `startupCommand`를 가진 인라인 템플릿
  적용 → 새 2페인 워크스페이스가 생성되고, 전환 시 스폰되며, 시작 커맨드가
  에코한 마커가 페인에 나타난다.
