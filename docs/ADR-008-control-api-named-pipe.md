# ADR-008: 네임드 파이프 기반 제어 API (M2.2)

상태: 채택 (2026-07-02)

## 배경

M2.1은 자동화 로직을 인프로세스에 두었다. 다음 단계(로드맵 §2.4b)는
terminal-f *외부*에 있어야 하는 로직 — 스케줄러, 또는 `claude -p`를 호출하는
LLM-in-the-loop 브로커 — 이 터미널 상태를 관찰하고 주입을 구동할 수 있게
한다. 이를 위해서는 외부 프로세스로 통하는 로컬 제어 채널이 필요하다.

## 결정

### TCP가 아닌 네임드 파이프

서버는 `interprocess` 크레이트로 Windows 네임드 파이프를 바인딩한다
(`GenericNamespaced` → `\\.\pipe\terminal-f-<id>.sock`).

- TCP localhost는 기각했다: 포트를 할당하고, 모든 로컬 프로세스가 접근할 수
  있으며, 잠재적인 네트워크 노출 위험이 있다. 네임드 파이프는 기본 DACL에
  의해 사용자 범위로 제한되며 포트가 필요 없다.
- 원시 Win32(`CreateNamedPipeW` 등)도 기각했다: 정확하지만 unsafe 표면이
  크다. `interprocess`는 블로킹 방식의 std 유사 `Read`/`Write` 스트림을
  제공하므로, 기존의 커넥션당 스레드 모델을 그대로 적용할 수 있다.

### 요청/응답만 지원 (서버 푸시 없음)

프로토콜은 줄바꿈으로 구분된 JSON이며, 한 줄에 요청 하나 → 응답 하나다.
실시간 출력을 원하는 브로커는 이벤트를 푸시받는 대신, 전진하는 바이트
오프셋으로 `readOutput`을 **폴링**한다.

- 이렇게 하면 각 커넥션이 단일 블로킹 read→dispatch→write 루프로 유지된다.
  두 번째 쓰기 스레드도 없고, 느린 브로커와 PTY 리더 사이의 배압(back-pressure)
  결합도 없다.
- 브로커가 어차피 소비하는 방식(커서를 앞으로 당기기)과 일치하며, 프론트엔드
  자체의 seq 기반 리플레이 모델과도 대응한다.

### 토큰 핸드셰이크가 접근 경계

시작 시 terminal-f는 무작위 토큰을 생성하고 사용자 범위 설정 디렉터리에
`control-api.json`(`{pipeName, token}`)을 기록한다. 커넥션의 첫 메시지는
반드시 `{method:"auth", params:{token}}`이어야 하며, 인증 전에는 다른 모든
메서드가 거부된다. 브로커는 그 파일을 읽어 파이프 이름과 토큰을 알아낸다.

- 보안 모델은 *로컬 사용자 신뢰*다: 해당 사용자로 실행되는 모든 프로세스가
  파일을 읽고 파이프를 구동할 수 있다. 이는 사용자 자신의 셸과 동일한 신뢰
  수준이며, 데스크톱 도구에 의도된 경계다. 더 강한 척하지 않고 그대로
  문서화했다.
- (UI에 노출되는) `control_api_info`는 파이프 이름과 파일 경로를 반환하지만,
  UI에 비밀 값이 드러나지 않도록 토큰은 반환하지 **않는다**.

### 파이프는 전송 수단이지 우회로가 아니다

API가 노출하는 모든 기능은 UI가 사용하는 것과 정확히 동일한 백엔드 경로를
재사용한다:

- `injectPrompt` → `do_inject` (킬 스위치, 페인별 주입 허용 목록(allowlist),
  유휴 게이트(idle gate), 브래킷 붙여넣기(bracketed paste), 감사 로그). 감사
  로그의 `source`는 `pipe:<client>`이므로, 어느 브로커가 행동했는지 로그에
  드러난다.
- `readOutput` → 새로운 페인별 `allowObserve` 플래그(스키마 v6, 기본값
  false)로 게이트되며, `allowInjection`과 대칭이다. 터미널 내용에는 비밀이
  담길 수 있으므로 관찰도 페인별 옵트인이다(로드맵 §2.7 #2).
- `listPanes` / `listWorkspaces` / `listRules` / `runRule`은 스토어/엔진을
  재사용한다.

게이트는 백엔드에 있으므로, 파이프 클라이언트는 UI보다 더 큰 권한을 가질 수
없다.

### 세션별 파일 스풀을 통한 출력 관찰

`readOutput`은 1 MiB 링 버퍼가 아닌 세션별 **스풀(spool) 파일**에서 제공되므로,
느리거나 재연결하는 브로커도 스풀 상한 내에서는 데이터를 잃지 않는다. 페인이
`allowObserve` 상태이면, 그 리더 스레드가 디코딩된 출력을
`<config>/spool/<sessionId>.log`에 추가한다. `readOutput{from, max}`는 바이트
`from`부터의 조각과 새 오프셋을 반환한다. 커서는 바이트 오프셋(byte offset)이다 —
seq→오프셋 매핑보다 단순하고 견고하다.

- 스풀은 상한이 있다(세션당 16 MiB): 가득 차면 이후 출력은 버려지고 파일은
  더 이상 커지지 않는다(브로커에게는 `total`이 멈춘 것으로 보인다). 상한을
  넘는 완전한 보존 아카이브가 아님을 명시적 한계로 둔다.
- 스풀은 세션 종료 시 제거된다. 관찰 이력은 페인 수명 동안만 존재하며 절대
  영속화되지 않는다.
- `allowObserve`가 아닌 페인은 스풀하지 않으며(비용 0), 해당 페인에 대한
  `readOutput`은 거부된다.

## 트레이드오프/한계

- 폴링 지연: 브로커는 자신의 폴링 주기에 맞춰 출력/diff를 본다. 리뷰/스케줄러
  브로커에는 수용 가능하다.
- 단일 사용자 로컬 신뢰만 지원한다. M2.2에는 브로커별 권한 범위 지정이 없다
  (인증된 모든 클라이언트가 옵트인된 모든 페인에 대해 모든 메서드를 호출할 수
  있다). 더 세밀한 ACL은 향후 작업이다.
- 앱 인스턴스당 하나의 파이프 인스턴스/이름을 가진다(이름은 실행마다
  무작위이며 정보 파일에 기록된다). 여러 앱 인스턴스는 각자 자기 것을 갖는다.
- `interprocess`는 기본 파이프 DACL을 사용한다. M2.2에서는 별도의 SACL/소유자
  강화가 없다(토큰이 1차 게이트다).

## 테스트

- `pipe::dispatch` 단위 테스트: 인증 게이트, 잘못된/올바른 토큰, 알 수 없는
  메서드, 잘못된 JSON, ping도 인증 필요.
- `tests/pipe_smoke.rs`: 실제 interprocess 리스너 + 클라이언트, 전체 인증
  핸드셰이크와 스텁 핸들러를 통한 메서드 왕복(전송 계층 증명).
- 스풀: 추가/오프셋 기준 읽기/상한/EOF/파일 없음에 대한 단위 테스트.
- E2E (인앱 autotest): `allowObserve` 게이트는 옵트인 전에 거부한다. 옵트인
  후 `readOutput`은 스풀에서 주입/에코된 내용을 반환한다.
- `injectPrompt` 라우팅은 `do_inject`를 재사용하며, 이는 이미 E2E 테스트를
  거쳤다(M2.0/M2.1).

## 참조 브로커

`examples/broker-git-review/` — Node 브로커: `control-api.json`을 읽고,
인증한 뒤, git 저장소를 폴링하고, 리뷰를 위해
`claude -p … --output-format json`을 호출한 다음, 결과를 라벨이 지정된
페인에 `injectPrompt`한다. "판단은 프로세스 밖에서, terminal-f는 눈과 손"을
시연한다. 예제일 뿐이며 앱이 실행하지 않는다.
