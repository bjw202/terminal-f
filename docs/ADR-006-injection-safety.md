# ADR-006: 프롬프트 주입 안전 게이트 (M2.0)

상태: 채택 (2026-07-02)

## 배경

자동화 비전(로드맵 §2: AI CLI 세션 간에 프롬프트를 중계하는 브로커 에이전트)은 PTY stdin에 대한 프로그래밍 방식의 쓰기를 필요로 한다. 이는 이 앱에서 가장 위험한 기능이다. 검증되지 않은 경로가 있다면 사용자의 셸에 임의의 커맨드를 입력할 수 있기 때문이다. M2.0은 어떤 자동화도 존재하기 *전에* 안전 장치를 먼저 구축하며, 수동으로만 트리거되는 `inject_prompt`만 출시한다.

## 결정

주입(injection)은 사용자 타이핑과 분리된 별도 경로이며, 백엔드에서 다층 게이트를 강제한다(프론트엔드와 향후 파이프 API 클라이언트는 이를 우회할 수 없다):

1. **킬 스위치** (`injection_paused`, 세션 전역): 켜져 있으면 모든
   `inject_prompt`가 거부된다. 사용자 타이핑(`write_pane`)은 어떤 경우에도 영향을 받지 않는다.
2. **대상 결정(target resolution)**: 명시적인 `paneId`, 또는 정확히 하나의 페인이 소유한
   `label`을 사용한다. 모호한 라벨은 후보 개수와 함께 거부된다 — 규칙은
   절대 추측해서는 안 된다. 라벨은 사용자가 지정하는 페인 메타데이터로(영속화, 스키마 v3),
   페인 id가 재시작 간에 유지되지 않는 임시 값이기 때문이다.
3. **페인별 허용 목록(allowlist)** (`allowInjection`, 기본값 **false**, 영속화):
   옵트인하지 않은 페인으로의 주입은 거부된다. 활성화된 동안 페인 헤더에 ⚡가
   표시된다.
4. **유휴 게이트(idle gate)**: 대상 세션이 1500 ms(`INJECT_IDLE_MS`) 동안
   아무 출력도 내지 않았어야 한다. 한창 출력을 생성 중인 TUI(답변 작성 중인
   Claude Code)에 주입하면 입력이 파괴적으로 뒤섞인다. 호출자는 명시적으로
   `requireIdle: false`를 전달할 수 있으나, 규칙(M2.1)에는 이것이 허용되지 않는다.
5. **브래킷 붙여넣기(bracketed paste)**: 백엔드는 각 세션의 출력 스트림에서
   DECSET/DECRST 2004를 추적하고(실제 터미널처럼), 대상 앱이 해당 모드를
   활성화한 경우에만 주입 텍스트를 `ESC[200~ … ESC[201~`로 감싼다 — 그래서
   여러 줄 프롬프트가 한 줄씩 스스로 실행되는 일이 없고, 모드를 사용하지 않는
   앱은 불필요한 이스케이프 바이트를 절대 받지 않는다. `submit`(끝의 `\r`)은
   명시적 플래그이며, 수동 커맨드에서는 기본값이 true다.
6. **감사 로그(audit log)**: 성공한 모든 주입은 JSON 한 줄을 추가한다
   (config.json 옆의 `audit.log`): 타임스탬프, 소스("manual"; 이후에는 규칙 id),
   워크스페이스/페인/세션, 바이트 수, submitted/bracketed 플래그,
   그리고 제어 문자를 이스케이프한 200자 미리보기. `read_audit`로 읽을 수 있다
   (팔레트: "Injection: Show audit log").

## 검토한 대안

- **`write_pane`에 플래그를 추가해 재사용**: 기각 — "raw" API를 호출하는 것만으로
  게이트를 건너뛸 수 없어야 한다. 모드를 가진 하나의 커맨드보다, 신뢰 수준이
  다른 두 개의 커맨드가 추론하기 더 단순하다.
- **수동 주입 시 확인 다이얼로그**: 불필요 — 수동 플로 자체가 사용자의
  확인이다. 기본 확인(confirm-by-default)은 채택된 로드맵 결정에 따라
  *규칙*(M2.1)에 적용된다.
- **항상 브래킷 붙여넣기로 감싸기**: 기각 — 모드 2004를 활성화한 적 없는 앱이
  `ESC[200~` 바이트를 입력으로 그대로 받게 된다.
- **프롬프트 패턴 매칭 기반 유휴 감지**: 더 정밀하지만 셸/도구별로 달라진다.
  출력 정지(quiescence) 기반은 도구에 무관하며 M2.0에는 충분하다.
  이 검사는 분리되어 있어 도구별 준비 상태 어댑터로 교체할 수 있다(M2.1+).

## 트레이드오프 / 알려진 한계

- 정지 ≠ 준비 완료: 작업 중에도 조용한 도구(CLI에서는 드물다)는 유휴 게이트를
  통과할 수 있다. 수용하기로 했다. 어댑터로 나중에 강화할 수 있다.
- 8 KB 읽기 경계에 정확히 걸쳐 분할된 DECSET 2004 시퀀스는 놓친다.
  앱이 프롬프트를 다시 그릴 때 모드를 재선언하므로 플래그는 스스로 복구된다.
- 감사 미리보기는 설계상 200자에서 잘린다(로그가 두 번째 무한 출력 저장소가
  되어서는 안 된다). 바이트 수는 항상 정확하다.
- 발동 빈도 제한(rate limit)과 루프 방지 쿨다운은 규칙 수준의 관심사로,
  규칙이 존재하는 M2.1로 미룬다.

## 향후 계획

M2.1 규칙 엔진은 `source = 규칙 id`와 기본 확인(confirm-by-default)으로 동일한
`inject` 경로를 호출한다. M2.2 파이프 API도 마찬가지다 — 게이트는 계속
백엔드 쪽에 남는다.
