# ADR-003: 출력 캡처와 시각 스냅샷의 분리

상태: 채택 (M0)

## 결정

서로 구별되는 두 저장소를 두며, 절대 혼용하지 않는다:

1. **원시 출력 캡처 (권위 있는 원본, 백엔드)**: 단조 증가하는 시퀀스
   번호(seq)가 붙은 UTF-8 청크들의 세션별 링 버퍼. 라이브 이벤트와
   워크스페이스 전환 리플레이의 공급원이다. 크기가 유한하며 (ADR-004),
   손실은 명시적으로 드러난다 (드롭 카운터 + UI 마커).
2. **시각 스냅샷 (프론트엔드, 외관용)**: `@xterm/addon-serialize`의 출력.
   언마운트/재마운트 후 터미널을 다시 그리는 용도로만 사용한다. 절대
   파싱하지 않고, 절대 영속화하지 않으며, 절대 출력 소스로 사용하지
   않는다.

## xterm serialize가 출력 캡처가 되어서는 안 되는 이유

- serialize는 *현재 버퍼 상태*로부터 이스케이프 시퀀스를 재구성한다.
  원래의 바이트 스트림(타이밍, 중간 갱신, 스크롤되어 사라진 시퀀스,
  alt-screen 전환)은 그것으로부터 복구할 수 없다.
- xterm 스크롤백과 우리의 스냅샷 상한에 의해 크기가 제한되어 — 소리 없이
  손실이 발생한다.
- Terminal 인스턴스가 마운트되어 있는 동안에만 존재한다. 우리의 비활성
  워크스페이스에는 Terminal 자체가 없다.
- 이것을 `term.write`로 되돌려 쓰는 것은 렌더링 기법이지 데이터 충실도가
  아니다. 자동화/워처(캡처된 출력의 M1+ 사용처)에 이것을 쓰면 재구성되고
  순서가 재배열된 데이터 위에 기능을 쌓는 셈이 된다.

## 검토한 대안

1. **serialize를 캡처로 사용** (전환할 때마다 스냅샷, 백엔드 버퍼 없음):
   위의 이유로 기각. 게다가 언마운트된 동안 생성된 출력을 캡처할 수 없다.
2. **완전한 원시 데이터 영속화 (파일 스풀)**: 진정한 충실도를 제공하지만
   M0에는 필요 없고 디스크 수명 주기 문제가 추가된다. 보류 (ADR-004의
   드롭 정책 참고).

## 트레이드오프

- 복원된 페인에는 이음새(seam)가 보일 수 있다: 스냅샷(시각적 재구성)
  뒤에 리플레이된 원시 바이트가 이어진다. 실제로는 셸이 프롬프트를 다시
  그리므로 M0에서는 허용 가능하다.
- 메모리: 페인마다 백엔드 링과 프론트엔드 스냅샷 문자열이 둘 다
  존재하지만, 둘 다 크기가 유한하다.

## 향후 계획

워처/자동화 기능은 반드시 백엔드 링(또는 그 후속인 파일 스풀)을 소비해야
하며, 직렬화된 스냅샷을 소비해서는 안 된다.
