# ADR-005: 라이브 PTY 소프트 캡과 메모리 정책

상태: 채택 (M0)

## 결정

| 제한 | 값 | 초과 시 |
|---|---|---|
| 워크스페이스 소프트 캡 | 8 | 생성은 성공하되 UI에 **경고**를 표시한다 |
| 워크스페이스 하드 캡 | 16 | 생성을 오류와 함께 **거부**한다 |
| 라이브 PTY 소프트 캡 | 32 | spawn을 명확한 오류와 함께 **거부**한다. 페인은 사유를 렌더링하고 세션 없는 상태로 남는다 |

추가로 **지연 스폰(lazy spawn)**: 세션은 해당 워크스페이스가 처음 활성화될
때 (또는 활성 워크스페이스 안에서 페인이 분할될 때)만 생성된다 — 시작
시점에 영속화된 모든 워크스페이스에 대해 미리 생성하는 일은 절대 없다.

"워크스페이스 16개 × 페인 16개 = 256"은 엄격히 *레이아웃* 용량으로만
취급한다 (트리와 설정이 그것을 지원한다는 의미). 라이브 PTY keep-alive의
목표가 **아니다**. ConPTY 세션 하나는 conhost.exe + 셸 프로세스 비용이
든다 (pwsh 인스턴스 하나가 유휴 상태에서 약 60–100 MB). 라이브 PTY
256개는 수 GB의 RSS에 더해 리더 스레드 256개를 의미한다 — 기술적으로는
스폰 가능하지만 데스크톱 앱으로서는 운영상 터무니없다.

세션당 메모리 정책: 유한 링 (1 MiB, ADR-004) + 고정 크기 스레드 스택.
페인당 프론트엔드: 워크스페이스가 마운트된 동안에만 5000줄 스크롤백을
가진 xterm 1개 (ADR-002), 언마운트 상태에서는 상한이 있는 스냅샷 문자열
1개. 따라서 백엔드 메모리는 O(라이브 세션 수), 프론트엔드 메모리는
O(활성 워크스페이스의 페인 수)다.

## 스펙이 제시한 옵션 vs. 우리의 선택

- 경고 → *워크스페이스 소프트 캡*에 채택 (저렴하고 비차단).
- 지연 스폰 → 모든 곳의 기본 스폰 전략으로 채택.
- 스폰 거부 → *라이브 PTY 소프트 캡*에 채택. 조용히 큐에 넣거나 다른
  세션을 자동으로 죽이는 것은 최소 놀람 원칙(least surprise)에 어긋나고,
  모달 방식의 사용자 확인 플로우는 M0에 없는 UI 범위이기 때문이다.
- 명시적 사용자 확인 → M1로 보류 (거부를 "이 세션에 한해 캡을 올릴지
  확인"으로 업그레이드할 수 있음).

## 트레이드오프

- 32-PTY 캡에 도달한 사용자는 다른 페인을 닫기 전까지 죽은 페인을 보게
  된다 — 눈에 보이고 사유도 설명되지만 자가 치유는 아니다 (M0에는 LRU
  세션 하이버네이션이 없다).
- 지연 스폰 때문에 복원된 워크스페이스로의 첫 전환은 셸 시작 비용을
  치른다 (pwsh 기준 약 수백 ms) — 수용한다. keep-alive 덕분에
  워크스페이스당 한 번만 치르는 비용이다.

## 향후 계획

M1: 하드 거부 대신 LRU 하이버네이션 (종료 후 활성화 시 재생성 표시).
`memory_stats`에 세션별 RSS 집계를 추가하여 캡을 개수 기반이 아닌 바이트
기반으로 전환한다.
