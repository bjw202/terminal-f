# ADR-010: 클립보드 이미지 붙여넣기 + 파일 드롭 브리지

상태: 채택 (2026-07-02, 실기기 테스트 후 같은 날 재설계)

## 배경

macOS에서는 Ctrl+V가 TUI에 도달하므로(그곳의 터미널 붙여넣기는 Cmd+V다)
Claude Code가 클립보드의 스크린샷을 직접 읽는다. Windows에서는 터미널이
Ctrl+V를 가로채 클립보드의 **텍스트**만 전달한다. 클립보드에 이미지
(Win+Shift+S 스크린샷)가 있으면 아무것도 전송되지 않고, TUI는 붙여넣기가
일어난 사실조차 알지 못한다. 사용자는 "스크린샷 → Ctrl+V → 이미지 첨부"와
"파일을 드롭 → 그 경로가 나타남"이 terminal-f에서 동작하기를 기대한다.

## 결정

TUI에 **파일 경로**를 건넨다 — 드래그 앤 드롭에서 이미 처리하는 형태다.
두 브리지 모두 브라우저 이벤트를 완전히 우회한다("실패했던 접근" 참조):

### Ctrl+V — 키를 가로채고 Rust에서 OS 클립보드를 읽는다

xterm.js는 Ctrl+V를 `^V`(0x16)로 매핑해 PTY로 보내고 keydown을 취소한다 —
그래서 브라우저는 Ctrl+V에 대해 **paste 이벤트를 절대 발생시키지 않는다**.
커스텀 키 이벤트 핸들러가 Windows Terminal처럼 xterm보다 먼저 Ctrl+V(그리고
Ctrl+Shift+V)를 가로채고 `paste_clipboard`를 호출한다:

- 백엔드가 `arboard`로 OS 클립보드를 직접 읽는다(WebView 권한 프롬프트도,
  이벤트 배관도 없다).
- **텍스트가 있으면 → 텍스트를 그대로 붙여넣는다**(`term.paste()` 경유이므로
  브래킷 붙여넣기(bracketed paste)가 여전히 적용된다). 텍스트와 이미지가 모두
  있으면 텍스트가 우선한다.
- **이미지만 있으면 → PNG로 인코딩**한다. RGBA 비트맵을 `png` 크레이트로
  인코딩해 `<config>/paste/img-<ms>.png`로 저장하고, 따옴표로 감싼 경로 + 뒤에
  공백 하나를 붙여넣는다 — Explorer에서 드롭한 파일과 정확히 같은 방식으로
  브래킷 붙여넣기로 감싼다. Claude Code가 이미지 경로를 감지해 첨부하는
  방식이 바로 이것이다.

트레이드오프: TUI가 더 이상 원시 `^V` 바이트를 받을 수 없다(예: vim
비주얼 블록 사용자는 리바인딩해야 한다) — Windows Terminal이 기본값으로
감수하는 것과 동일한 트레이드다.

### 파일 드롭 — Tauri의 drag-drop 이벤트를 소비한다

Tauri(기본값인 dragDropEnabled)는 OS 파일 드롭을 DOM보다 먼저 가로채므로,
HTML5 `drop`은 절대 발생하지 않는다. 프론트엔드는
`getCurrentWebview().onDragDropEvent`를 구독하는데, 이는 **실제 OS 경로**
(브라우저 `File`에는 경로가 없다)와 드롭 위치를 제공한다. 커서 아래의
페인은 `elementFromPoint`로 결정하고(물리 픽셀 → CSS 픽셀 변환은
devicePixelRatio 사용), 따옴표로 감싼 경로들을 그 위치에 붙여넣는다.

### DOM paste 리스너 (폴백으로 유지)

페인마다 있는 캡처 단계(capture phase) `paste` 리스너는 브라우저가 이미지
데이터를 담은 paste 이벤트를 실제로 *전달하는* 모든 경로(예: 프로그래밍
방식 또는 컨텍스트 메뉴 붙여넣기)를 여전히 처리한다: 이미지만 있는 클립보드
→ `save_pasted_image`(IPC를 통한 base64) → 경로 붙여넣기. 텍스트가 있는
이벤트는 기본 경로를 유지한다.

### 실패했던 접근 (그리고 E2E가 왜 못 잡았나)

첫 구현은 브라우저 paste 이벤트에만 의존했다. E2E는 통과했다 — **합성**
ClipboardEvent를 디스패치했기 때문인데, 이는 바로 문제가 되는 지점을
우회한다: xterm이 실제 Ctrl+V keydown을 취소하므로 실제 paste 이벤트는
존재하지 않는다. 실기기 테스트에서 실패했고, 브리지는 위의 키 가로채기 +
Rust 클립보드 설계로 재작업되었다. 기록해 둔 교훈: 합성 입력 이벤트는
리스너 로직을 검증할 뿐, 키 전달 경로를 검증하지 못한다.

### 제한

- 붙여넣기 디렉터리는 최근 이미지 20개만 유지한다(`PASTE_KEEP`). 저장할
  때마다 정리된다. 같은 밀리초의 붙여넣기는 카운터 접미사가 붙는다.
  32 MiB 초과는 거부된다.
- 파일은 재시작 후에도 남는다(TUI가 나중에 경로를 읽을 수 있으므로).
  상한이 디스크 사용량을 제한한다.

## 테스트

- 단위 (`paste.rs`): 저장/정리/충돌/mime; `encode_png` 왕복(디코딩해 픽셀
  비교)과 크기 불일치 거부; **실제 OS 클립보드**에 대한 `read_clipboard`
  (텍스트를 설정하고, Text가 우선함을 단언하고, 사용자의 클립보드를 복원).
- E2E (autotest): 합성 이미지 paste 이벤트 → 셸 버퍼에 경로(DOM 폴백 체인);
  공백이 포함된 경로로 `pastePathsToPane` → 버퍼에 따옴표로 감싼 경로(드롭
  체인); `paste_clipboard` IPC 왕복(읽기 전용 — 라이브 클립보드 텍스트를
  셸에 붙여넣으면 뜻하지 않은 개행이 실행될 수 있다).
- 수동: Win+Shift+S → Claude Code 페인에 Ctrl+V; Explorer에서 파일을 페인에
  드롭.
